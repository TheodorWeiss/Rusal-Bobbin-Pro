# Техническая спецификация модели раскроя (v2.0)

## 1. Назначение

Модель выполняет распределение заказов по бобинам и размещение их на полотне с учётом геометрических и технологических ограничений.

Результат — единый JSON-файл с координатами всех размещённых заказов, сгруппированных по бобинам, с метриками эффективности.

---

## 2. Входные данные

Модель принимает два файла.

### 2.1. Таблица заказов

Источник: CSV / Excel / JSON.

Поля:

| Поле | Тип | Описание |
|---|---|---|
| order_id | string | Уникальный ID заказа |
| width_mm | int | Ширина заказа (мм), диапазон 175–1340 |
| length_m | int | Длина заказа (м), диапазон 25–480 |
| alloy | string | Марка сплава (1200, 8011) |
| thickness_um | float | Толщина (мкм) |
| priority | int | Очередь (1, 2, 3, ...) |

Инварианты:
- один заказ = один прямоугольник (заказ не делится)
- order_id уникален во всём наборе данных
- width_mm > 0, length_m > 0

Примечание: в текущем датасете колонка «Количество» отсутствует — каждая строка представляет один заказ-прямоугольник.

### 2.2. Справочник межкройного реза

Источник: CSV / JSON.

Поля:

| Поле | Тип | Описание |
|---|---|---|
| alloy | string | Марка сплава |
| thickness_um | float | Толщина (мкм) |
| inter_cut_mm | int | Ширина межкройного реза (мм) |

Инвариант: для каждой пары (alloy, thickness_um), встречающейся в таблице заказов, должна существовать запись в справочнике.

---

## 3. Геометрия бобины

Фиксированные параметры (для хакатона):

| Параметр | Значение |
|---|---|
| Ширина | 1500 мм |
| Длина | 10 000 м |
| Кромка (edge_trim_mm) | 15 мм с каждой стороны |
| Полезная ширина | 1470 мм (1500 − 2×15) |

Ключевое правило: **одна бобина = один alloy + одна thickness_um**. Заказы разных сплавов или толщин не могут размещаться на одной бобине.

---

## 4. Координатная система

Каждый размещённый заказ описывается четырьмя координатами:

```
x_start_mm  — начало по ширине бобины (ось X: 0 → 1500)
y_start_m   — начало по длине бобины (ось Y: 0 → 10000)
width_mm    — ширина заказа
length_m    — длина заказа
```

Заказ занимает прямоугольник `[x_start, x_start + width] × [y_start, y_start + length]`.

Заказы не поворачиваются — ширина всегда вдоль оси X (ширина бобины), длина всегда вдоль оси Y (длина бобины).

---

## 5. Геометрические ограничения

### 5.1. Кромка (edge_trim)

Применяется только по оси X:

```
x_start_mm ≥ edge_trim_mm                          (левая кромка)
x_start_mm + width_mm ≤ bobbin_width_mm − edge_trim_mm   (правая кромка)
```

По оси Y кромка не применяется.

### 5.2. Границы по длине

```
y_start_m ≥ 0
y_start_m + length_m ≤ bobbin_length_m
```

### 5.3. Отсутствие пересечений

Для любых двух заказов на одной бобине: если есть перекрытие по X **и** перекрытие по Y — это ошибка (наложение).

---

## 6. Межкройный рез (inter_cut)

### 6.1. Определение

Минимальное расстояние между двумя заказами по оси X, которые одновременно находятся под ножами стана (т.е. имеют перекрытие по Y).

### 6.2. Когда применяется

Если два заказа на одной бобине:
- **имеют перекрытие по Y** (их y-диапазоны пересекаются), и
- **не пересекаются по X**,

то расстояние между ними по X (зазор) должно быть ≥ inter_cut_mm:

```
gap_x = x_start_B − (x_start_A + width_A)    # при x_start_A < x_start_B
gap_x ≥ inter_cut_mm
```

### 6.3. Когда НЕ применяется

- К одиночным заказам (нет соседа).
- К краям бобины (учтено в edge_trim).
- **По оси Y** — между заказами, не имеющими y-перекрытия, зазор по X не требуется.

Обоснование: если два заказа не пересекаются по Y, они режутся с разными настройками ножей (в разных «проходах» стана). Физически между ними нет одновременно работающего ножа, поэтому технологический зазор не нужен.

---

## 7. Группировка по бобинам

Перед размещением модель разбивает заказы на группы по ключу (alloy, thickness_um). Каждая группа размещается на своих бобинах независимо от остальных групп.

Количество бобин для каждой группы определяется алгоритмом в процессе размещения (теоретический минимум — `ceil(суммарная эффективная площадь / площадь бобины)`).

---

## 8. Очереди, приоритеты и спутники

### 8.1. Очередность

У каждого заказа есть `priority` (1, 2, 3, ...). Младший номер = высший приоритет.

### 8.2. Глобальное правило размещения

Строгое правило (из ТЗ РУСАЛа): **пока не размещены ВСЕ заказы очереди N (во всём плане, по всем бобинам данной группы), алгоритм не имеет права переходить к самостоятельному (main) размещению заказов очереди N+1.**

Это значит: если хотя бы один заказ очереди 1 ещё не размещён, ни на одной бобине не может появиться main-заказ очереди 2.

### 8.3. Типы заказов: main и satellite

**main** — заказ, ради которого открыта полоса на бобине. Определяет длину полосы (высоту «ряда»). На каждой бобине `main_priority` — это минимальный priority среди всех main-заказов на этой бобине.

**satellite** — заказ, заполняющий свободное место рядом с main-заказом по ширине.

### 8.4. Правило спутников

**Базовое правило:** спутник может быть только из очереди `main_priority + 1`.

Пример: если main-заказ имеет priority=1, рядом с ним может стоять спутник только из очереди 2. Спутник из очереди 3 рядом с main очереди 1 — **запрещён**.

**Исключение:** если на данном листе (бобине) **все main-заказы уже полностью размещены** и остаётся свободное место, допускается размещение спутников из очереди `main_priority + 2`.

Пример: бобина с main_priority=1. Все оставшиеся main-заказы очереди 1 уже размещены на этом листе. В хвосте бобины остаётся 200 м свободной длины. В эту область можно разместить main-заказы очереди 2, а к ним — спутники из очереди 3 (satellite_2).

### 8.5. Формат поля type

```
type = "main" | "satellite_1" | "satellite_2"
```

Формула: `satellite_level = priority − main_priority`, где `main_priority` — минимальный priority среди main-заказов **в данной полосе** (ряде по Y).

| priority заказа | main_priority полосы | type |
|---|---|---|
| 1 | 1 | main |
| 2 | 1 | satellite_1 |
| 2 | 2 | main |
| 3 | 2 | satellite_1 |
| 3 | 1 | satellite_2 (только при исчерпании main на листе) |

---

## 9. Ограничения по материалу

В пределах одной бобины:
- все заказы имеют одинаковый alloy
- все заказы имеют одинаковый thickness_um
- inter_cut_mm определяется из справочника по этой паре

## 10. Выходной JSON

Модель возвращает **один JSON-файл** с иерархической структурой: план → бобины → заказы.

```json
{
  "instruction_metadata": {
    "batch_id": "PLAN-2026-04-11-001",
    "timestamp": "2026-04-11T10:00:00Z",
    "factory": "САЗ",
    "machine_id": "MILL-05"
  },
  "bobbins": [
    {
      "bobbin_id": "B-1200-12.0-001",
      "source_material": {
        "alloy": "1200",
        "thickness_um": 12.0,
        "bobbin_width_mm": 1500,
        "bobbin_length_m": 10000
      },
      "layout_configuration": {
        "inter_cut_mm": 7,
        "edge_trim_mm": 15
      },
      "cutting_map": [
        {
          "order_id": "ORD-100004",
          "type": "main",
          "priority": 1,
          "coordinates": {
            "x_start_mm": 15,
            "y_start_m": 0,
            "width_mm": 600,
            "length_m": 120
          }
        },
        {
          "order_id": "ORD-100010",
          "type": "satellite_1",
          "priority": 2,
          "coordinates": {
            "x_start_mm": 622,
            "y_start_m": 0,
            "width_mm": 400,
            "length_m": 80
          }
        }
      ],
      "efficiency_metrics": {
        "total_used_area_m2": 104.0,
        "waste_area_m2": 14896.0,
        "waste_percentage": 99.31
      }
    }
  ],
  "summary": {
    "total_bobbins": 7,
    "total_orders": 450,
    "total_useful_area_m2": 12000.0,
    "total_bobbin_area_m2": 105000.0,
    "overall_waste_percentage": 88.6
  }
}
```

### Обоснование иерархической структуры (вместо плоского placements[])

Каждая бобина — самодостаточный объект, содержащий:
- `source_material` — какой сплав и толщина (не дублируется в каждом заказе)
- `layout_configuration` — inter_cut_mm и edge_trim_mm (валидатор берёт их отсюда, не ищет в шапке)
- `cutting_map` — массив размещённых заказов с координатами
- `efficiency_metrics` — метрики именно этой бобины

Преимущества:
- визуализатор берёт одну бобину → рисует → переключается на следующую
- inter_cut_mm разный для разных пар (сплав, толщина) и лежит рядом с заказами
- bobbin_id явный — оператор видит «лист B-1200-12.0-003»
- метрики считаются и на уровне бобины, и на уровне всего плана (summary)

---

## 11. Метрики эффективности

Метрики — **ключевой критерий оценки** (40% итоговой оценки на хакатоне).

### 11.1. На уровне бобины

```
total_used_area_m2 = Σ (width_mm / 1000) × length_m    для всех заказов на бобине
waste_area_m2      = (bobbin_width_mm / 1000) × bobbin_length_m − total_used_area_m2
waste_percentage   = waste_area_m2 / ((bobbin_width_mm / 1000) × bobbin_length_m) × 100
```

Примечание: `waste_area_m2` включает кромки, межкройные резы и неиспользованный «хвост» бобины.

### 11.2. На уровне плана (summary)

```
total_useful_area_m2  = Σ total_used_area_m2 по всем бобинам
total_bobbin_area_m2  = количество_бобин × 15000 м²
overall_waste_percentage = (total_bobbin_area_m2 − total_useful_area_m2) / total_bobbin_area_m2 × 100
```

### 11.3. Формула из ТЗ РУСАЛа (для оценки)

```
Доля обрезков = (Σ площадь межкройных резов + Σ площадь неиспользованного металла на начатых бобинах)
               / Σ площадь всех использованных бобин
```

Визуализатор должен пересчитывать метрики самостоятельно из cutting_map и сравнивать с заявленными в JSON — при расхождении > 0.1% помечать как предупреждение.

---

## 12. Критерии корректности результата

JSON считается корректным, если выполнены **все** следующие условия.

### 12.1. Геометрия (проверяется для каждой бобины)

- Все заказы находятся внутри бобины (кромки, границы по длине).
- Нет пересечений (overlap) между любыми двумя заказами.
- Соблюдён inter_cut_mm между заказами с y-перекрытием.

### 12.2. Данные

- Все order_id уникальны в пределах всего плана (не только бобины).
- width_mm > 0, length_m > 0 для каждого заказа.
- Координаты неотрицательны.
- alloy и thickness_um всех заказов на бобине совпадают с source_material бобины.

### 12.3. Логика приоритетов (проверяется глобально по всему плану)

- **Полнота:** все заказы из входной таблицы размещены ровно один раз (ни пропущенных, ни дублей).
- **Порядок очередей:** если в плане есть main-заказ очереди N+1, значит все заказы очереди N (по всему плану, по всем бобинам данной группы) уже полностью размещены.
- **Спутники:** satellite_1 имеет priority = main_priority_полосы + 1. satellite_2 допускается только если все main-заказы на этом листе уже размещены, и имеет priority = main_priority_полосы + 2.
- **Материал:** все заказы на одной бобине имеют одинаковые alloy и thickness_um.
